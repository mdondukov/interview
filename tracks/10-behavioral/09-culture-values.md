## "Why do you want to work here?"

### Framework

```
┌─────────────────────────────────────────────────────────────────────┐
│                     Why This Company                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Structure your answer around 3 areas:                              │
│                                                                     │
│  1. MISSION / PRODUCT                                               │
│     - What problem do they solve?                                   │
│     - Why does it matter to you?                                    │
│     - Personal connection if authentic                              │
│                                                                     │
│  2. TEAM / CULTURE                                                  │
│     - What have you heard about the culture?                        │
│     - What attracts you to how they work?                           │
│     - Who have you talked to?                                       │
│                                                                     │
│  3. GROWTH / CHALLENGE                                              │
│     - What will you learn here?                                     │
│     - What technical challenges excite you?                         │
│     - How does this fit your career trajectory?                     │
│                                                                     │
│  Research required:                                                 │
│  - Company mission, values, recent news                             │
│  - Engineering blog, tech talks                                     │
│  - Glassdoor, LinkedIn - but verify                                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Пример ответа

```
"I'm excited about this role for three reasons:

First, the mission. I've been using [Product] personally for years.
Financial literacy is something I care about - I grew up in a
family where money management wasn't taught, and I've seen how
tools like yours can change people's lives. Working on something
I believe in matters to me.

Second, the technical challenges. You're handling millions of
transactions daily with regulatory requirements - that intersection
of scale and compliance is exactly what I want to grow in. I read
your engineering blog post about the distributed transaction system,
and that kind of problem is fascinating to me.

Third, the culture. I've spoken with two engineers here - they
both mentioned the emphasis on mentorship and the blameless
postmortem culture. Those are things I've advocated for in my
current role, and I'd love to be in an environment where they're
already valued.

I'm at a point in my career where I want to work on meaningful
problems at scale, and this role seems like the right fit."
```

## "Why are you leaving your current job?"

### Framework

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Talking About Leaving                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  DO:                                                                │
│  - Focus on what you're moving TOWARD, not away from                │
│  - Be honest but professional                                       │
│  - Show gratitude for what you learned                              │
│  - Explain growth trajectory                                        │
│                                                                     │
│  DON'T:                                                             │
│  - Badmouth current employer                                        │
│  - Complain about people                                            │
│  - Sound desperate to leave                                         │
│  - Lie (small world, will catch up)                                 │
│                                                                     │
│  Safe themes:                                                       │
│  - Ready for next challenge                                         │
│  - Want to grow in area X (company doesn't offer)                   │
│  - Interested in different domain/scale                             │
│  - Company direction changed                                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Пример ответа

```
"I've had a great 4 years at [Company]. I joined when we were
50 engineers, grew to lead a team, and shipped products I'm proud
of. I've learned a lot, especially about building from scratch.

I'm looking to move for two reasons:

First, I want to work at larger scale. We're processing thousands
of requests per second, but I'm interested in millions. Your
infrastructure challenges are the next level of what I want to
learn.

Second, I've grown as much as I can in my current role. There's
no Staff engineering path at my company - the next step would be
management, which isn't where I want to go. Here, I see a clear
IC progression that aligns with my career goals.

I'm not running from anything - I'm still learning and contributing.
But I'm ready for a bigger challenge, and this seems like the
right opportunity."
```

## "Where do you see yourself in 5 years?"

### Framework

```
Don't:
- "I want your job" (cliché, sounds political)
- "I don't know" (sounds like no ambition)
- "Managing a team" (unless you mean it)

Do:
- Show ambition AND realism
- Align with role/company trajectory
- Show you've thought about it
- Be authentic
```

### Пример ответа

```
"In 5 years, I want to be a technical leader who's had significant
impact on the company's direction.

More specifically, I see three areas of growth:

Technical depth: I want to be the go-to person for a domain -
maybe distributed systems or data engineering. Someone teams
come to for architectural guidance.

Technical breadth: I want to understand how the whole system
fits together, not just my corner. I find the intersection of
business and technology fascinating.

Impact through others: By then, I expect to have mentored
several engineers who've grown into senior roles. My contribution
should be multiplicative, not just additive.

I'm deliberately NOT saying 'management' because I want to stay
on the IC track. What excites me is solving hard problems and
helping others solve them, not managing schedules and budgets.

This role fits because it's technical leadership at scale, which
is exactly the experience I need to get there."
```

## "What are your strengths?"

### Framework

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Talking About Strengths                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Structure:                                                         │
│  1. State the strength                                              │
│  2. Provide evidence                                                │
│  3. Connect to the role                                             │
│                                                                     │
│  Good strengths to discuss:                                         │
│  - Technical: specific skill, not just 'good coder'                 │
│  - Problem-solving: how you approach challenges                     │
│  - Communication: explaining complex things                         │
│  - Collaboration: working with different types of people            │
│  - Learning: picking up new things quickly                          │
│                                                                     │
│  Avoid:                                                             │
│  - Generic answers ('I work hard')                                  │
│  - Humble brags disguised as strengths                              │
│  - Strengths irrelevant to the role                                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Пример ответа

```
"I'd highlight three strengths:

First, debugging complex systems. When something's broken and
nobody knows why, I'm the person who gets called. In my current
role, I've resolved production issues that had stumped the team
for days. I have patience for tracing through logs and a mental
model for how systems fail.

Second, translating between technical and non-technical. I'm often
the person who explains to product why something is hard, or
explains to engineers why the business requirement makes sense.
I've led several cross-functional projects where this skill was
critical.

Third, learning new domains quickly. In the last 3 years, I've
worked in fintech, healthcare, and e-commerce. Each time, I was
productive within weeks. I have a process for identifying what
matters most to learn first.

For this role, I think the debugging and cross-functional
communication strengths are most relevant given the distributed
systems challenges and stakeholder complexity you've described."
```

## "What are your weaknesses?"

### Framework

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Talking About Weaknesses                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Good answers:                                                      │
│  - Real weakness (shows self-awareness)                             │
│  - What you're doing about it (shows growth mindset)                │
│  - Progress you've made (shows it's manageable)                     │
│  - Not fatal for the role                                           │
│                                                                     │
│  Bad answers:                                                       │
│  - "I'm a perfectionist" (cliché, humble brag)                      │
│  - "I work too hard" (not a real weakness)                          │
│  - Something critical for the role                                  │
│  - "I don't have weaknesses"                                        │
│                                                                     │
│  Safe areas:                                                        │
│  - Public speaking (improvable, rarely critical)                    │
│  - Specific technology gap (not core to role)                       │
│  - Delegation (common, relatable)                                   │
│  - Saying no (common, relatable)                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Пример ответа

```
"Something I've been working on is delegation. Early in my career,
I defaulted to 'I'll just do it myself' because it felt faster.
But as I've grown more senior, I've realized that doesn't scale
and doesn't help the team grow.

Specifically, I'd take on tasks instead of letting junior engineers
struggle through them. I was optimizing for short-term speed, not
long-term team capacity.

What I've done about it:
- I now explicitly ask 'Who else could do this?' before taking
  something on
- I've started tracking how much I delegate as a personal metric
- When I do take something, I try to pair with someone so it's
  a learning opportunity

It's still something I work at. When pressure is high, my instinct
is still to do it myself. But I've gotten much better - my last
two projects, I did about 30% of the coding, which was the right
amount for a lead role.

I don't think this will be a problem for this role - actually,
the scale here would force good delegation habits."
```

## "Tell me about yourself"

### Framework (2-3 minutes max)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Tell Me About Yourself                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Structure:                                                         │
│  1. Current role (30 sec) - what you do now                         │
│  2. Past highlights (60 sec) - relevant experience                  │
│  3. Future direction (30 sec) - why you're here                     │
│                                                                     │
│  Tips:                                                              │
│  - Not a biography - curated highlights                             │
│  - Emphasize what's relevant to THIS role                           │
│  - End with why you're excited about this opportunity               │
│  - Practice until it sounds natural, not rehearsed                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Пример ответа

```
"I'm currently a Senior Engineer at [Company], where I lead
backend development for our payments platform. My team of 5
engineers handles about 500K transactions daily.

Before that, I spent 3 years at a startup where I was one of
the first engineers. I got to build systems from scratch, make
architectural decisions, and grow with the company from 5 to 50
engineers. That experience taught me how to work with ambiguity
and wear many hats.

Earlier in my career, I did a couple years at a large company,
which gave me exposure to enterprise-scale systems and formal
engineering practices.

What threads through my career is building reliable systems that
handle real money - I've worked in fintech for most of my career
because I find the combination of technical challenge and real-world
impact compelling.

I'm talking to you because I'm ready for larger scale challenges.
You're processing orders of magnitude more transactions, with more
complex distributed systems problems. That's exactly the next
chapter I'm looking for."
```

## "Do you have any questions for us?"

### Framework

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Questions to Ask                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  About the role:                                                    │
│  - "What does success look like in 6 months?"                       │
│  - "What's the biggest challenge I'd face?"                         │
│  - "What's the team's biggest pain point right now?"                │
│                                                                     │
│  About the team:                                                    │
│  - "How does the team make technical decisions?"                    │
│  - "What's the code review process like?"                           │
│  - "How do you handle on-call?"                                     │
│                                                                     │
│  About growth:                                                      │
│  - "What does the engineering ladder look like?"                    │
│  - "How do people learn and grow here?"                             │
│  - "What's the path to Staff/Principal?"                            │
│                                                                     │
│  About culture:                                                     │
│  - "What's something you'd change about working here?"              │
│  - "How do you handle disagreements?"                               │
│  - "What type of person thrives here?"                              │
│                                                                     │
│  Show you've done research:                                         │
│  - "I read about X in your blog. Can you tell me more?"             │
│  - "I saw you recently launched Y. How did that go?"                │
│                                                                     │
│  DON'T ask about:                                                   │
│  - Salary (too early in process, unless they bring it up)           │
│  - Vacation first (seems like you don't want to work)               │
│  - Things easily found on website                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Good Questions Examples

```
"What's the biggest technical challenge the team is facing right now?"

"How do you balance feature development with technical debt?"

"Can you tell me about a recent architectural decision and how it was made?"

"What does the on-call rotation look like? What's a typical incident?"

"I noticed you use [technology]. How's that working at your scale?"

"What's something that surprised you about working here?"

"How do you think about the trade-off between shipping fast and code quality?"

"What would make someone successful vs. unsuccessful in this role?"
```

## Red Flags

```
❌ Mistakes to avoid:

1. Generic answers
   "I want to work somewhere innovative"
   → Be specific about THIS company

2. Obviously didn't research
   "So what do you guys do exactly?"
   → Should know basic facts

3. Only about money/title
   "What's the fastest path to promotion?"
   → Should show genuine interest

4. Badmouthing previous employer
   "My last company was terrible"
   → Shows poor judgment

5. No questions prepared
   "I think you covered everything"
   → Shows lack of engagement

6. Inauthentic answers
   Obvious rehearsed responses that don't feel real
   → Be genuine
```
